## Why v5.1

v5 EBT holds a `_registeredMeters` map and gates `settle()` on it. Owner-only registry mutation. Three problems:

1. State grows with meter count â†’ ZK proof costs grow over time.
2. Every meter add/remove is a separate on-chain tx.
3. Single-key compromise of the owner enables fake meter registrations.

v5.1 removes the on-chain meter list entirely. Settlement integrity is anchored in two layered signatures:

- **HAT signature** over the session payload (proves authentic hardware reading)
- **Meter Authority signature** over the HAT pubkey (proves the HAT is currently approved)

The Meter Authority pubkey is the only meter-related state the contract holds. Constant size regardless of meter count.

## State (constant size)

```
_initialized: Boolean
_owner: Either<ZswapCoinPublicKey, ContractAddress>
_meterAuthorityPubkey: Bytes<32>
_balances: Map<Bytes<32>, Uint<128>>           (public balances; same as v5)
_totalSupply: Uint<128>
settlementCount: Counter
_settledSessions: Set<Bytes<32>>               (replay protection)
_bps{Producer,Operations,Dividend,Dao}: Uint<16>
_{operationsRecipient,dividendRecipient,daoRecipient}: Bytes<32>
```

The only growth term is `_settledSessions` (32 bytes per settled session). At thousands of sessions per producer per year, this is far smaller than v5's per-meter state would be at the same scale.

## Settle flow

```
settle(
  // session payload
  sessionID, hatPubkey, amount, expiresAt,
  // recipient
  producer,
  // pre-computed slice amounts
  producerAmt, opsAmt, divAmt, daoAmt,
  // signatures
  hatSig, authoritySig,
):
  assertInitialized
  assert amount > 0
  assert blockTimeLte(expiresAt)            // chain-anchored freshness check
  assert sessionID not in _settledSessions  // Replay protection
  assert producerAmt + opsAmt + divAmt + daoAmt == amount

  authMessageHash = persistentHash([hatPubkey, expiresAtBytes])
  assert signature_valid(_meterAuthorityPubkey, authMessageHash, authoritySig)

  sessionPayloadHash = persistentHash([sessionID, hatPubkey, amountBytes])
  assert signature_valid(hatPubkey, sessionPayloadHash, hatSig)

  // Apply the mint to public balance map (4 recipients)
  _balances[producer] += producerAmt
  _balances[opsRecipient] += opsAmt
  ...
  _totalSupply += amount
  _settledSessions.insert(sessionID)
  settlementCount.increment()
```

**On the freshness check:** the contract uses Compact's stdlib
`blockTimeLte` (from `kernel`) which compares `expiresAt` against the
chain's actual block time. An earlier draft accepted `currentTime` as a
circuit parameter and asserted `currentTime < expiresAt`, which was
unsound â€” a malicious caller could pass `currentTime = 0` and bypass
the expiry check entirely. `blockTimeLte` anchors the verifier's clock
to the chain's, which the caller cannot manipulate.

**Critical property**: there is no permission check on the *caller*. Anyone can submit a settle() â€” what matters is that they have valid signatures from a HAT and the Authority. PollPower's settlement-relayer is no longer privileged; it just happens to be the entity that knows how to acquire and forward valid attestations.

**Slice policy enforced in-circuit (caveat I-2 fixed).** The contract
verifies that producerAmt/opsAmt/divAmt/daoAmt match the bps policy in
state, not just that they sum to amount. Without this, a permissionless
caller with valid HAT+Authority sigs could skew slices arbitrarily
(e.g., 100% to producer, 0% to ops/div/dao), bypassing the protocol
margin policy. The check uses cross-multiplication (`slice * BPS_TOTAL
âˆˆ (amount * bps_i - BPS_TOTAL, amount * bps_i + BPS_TOTAL)`) to avoid
Compact's lack of integer division. The Â±1 base unit tolerance
accommodates floor-rounded off-chain computations like the v5
convention `producerAmt = amount - opsAmt - divAmt - daoAmt`.

This closes the v5â†’v5.1 trust gap. v5 trusted PollPower's settlement-
relayer to compute slices correctly; v5.1 enforces the policy at the
contract layer regardless of caller.

- `setMarginPolicy`, `setRecipient`, `transferOwnership` â€” not in this draft. Can be added in v5.2 if needed; for the pilot we use deploy-time defaults and rotate the contract for any policy change.

## Threat analysis

| Compromise | Damage | Recovery |
|---|---|---|
| Settlement-relayer wallet | None â€” can't forge signatures | No-op |
| Single HAT secure element | That meter's readings spoofed | Council removes HAT pubkey from approved set; meter re-installs |
| Meter Authority key | Bad attestations until expiresAt | Owner ceremony-rotates authority; in-flight bad attestations expire |
| Owner key | Authority can be rotated to attacker key | Wrap owner with multi-sig (v6 pattern) before mainnet |
| Threshold council | Full takeover | Standard multi-sig majority risk |

The defining property: **single-key compromise of any individual party does not enable forged settlement.**
