# EBT v7.1 — Living Dividend Integration Diff

**Date:** 2026-07-01
**Status:** ✅ Written, syntax-clean on compactc 0.31.0. See
[`ebt-v7.1.compact`](./ebt-v7.1.compact).
**Base:** EBT v7 (deployed Preview `667d7f2aad9fac8613604df544d608ee2956f1771e440cc0c666592e80bec2b4`)

## Two things v7.1 adds

1. **Ledger log** on dividend-slice claims so the LD keeper can subscribe.
2. **Multisig-gated LD-binding setters** so no single key can bind or unbind
   the LD pointer — 3-of-5 ring signatures required, same helper as the
   LD contract itself uses.

## Implementation notes

### 1. Ledger log instead of `emit`

Original design proposed `emit DividendMinted { ... }` using MIP-0002
events. compactc 0.31.0 exposes `emit` for stdlib-declared event types
(`UnshieldedMint`, `ShieldedSpend`, etc.) but does not yet accept
user-declared event types (the MIP-0002 e2e test at
midnightntwrk/midnight-js @ 2026-06-30 only exercises stdlib events).

v7.1 writes to `_dividendMintedLog: Map<Uint<64>, DividendMintedEntry>`
instead. Keepers subscribe to the map's growth via the indexer using
`subscribeToContractActionEvents` (see `living-dividend/ld-keeper.ts`),
semantically equivalent for the LD downstream. Upgrade path: swap the
ledger write for `emit` once compactc exposes user-event types.

### 2. Log write in `claimSplit(kind=1)`, not `settle()`

v7 records the dividend slice as owed in `_pendingDiv` at settle time
and only actually mints during `claimSplit(1)`. LD only cares about the
moment EBT lands at the recipient, so the log entry belongs on the
mint edge.

### 3. Multisig gating on setters (added 2026-07-01)

Original v7.1 design had `assertOnlyOwner()` gating `setLivingDividendAddress`
and `clearLivingDividendAddress`. That means a single owner-key holder
could bind or unbind LD unilaterally, which:

- Makes LD activation NOT a ceremony — it's a single-key op.
- Doesn't exercise the multisig code path on Preview, so we'd get
  zero rehearsal for the real ring ceremony.

**Change:** both setters now require a 3-of-5 multisig bundle over the
signed payload, verified via a new `multisig_signature_valid` witness
that delegates to the shared `verifyMultisigBundle` helper. The owner
key is neither necessary nor sufficient for LD binding.

The owner still gates producer attestation, meter authority updates,
ownership transfer, split claims, and manual reissue — same as v7. Two
separate authority layers now:

- **Owner (single key)** — operational: attest producers, set meter
  authority, transfer ownership, claim splits, manual reissue.
- **Multisig authority** — governance: bind/unbind LD.

### 4. `initialize()` signature change

One new parameter: `multisigAuthorityHash: Bytes<32>` — the SHA-256
of sorted admin ring pubkeys. Same helper (`verifyMultisigBundle`)
computes this off-chain during deploy.

Updated signature:

```
initialize(
  meterAuthorityPubkey:  Bytes<32>,
  multisigAuthorityHash: Bytes<32>,  // NEW in v7.1
  operationsRecipient:   UserAddress,
  dividendRecipient:     UserAddress,
  daoRecipient:          UserAddress,
)
```

The deploy script updates: compute the multisig authority hash from
the pilot-mock ring pubkeys (or real Tangem ring pubkeys post-registration)
and pass it into `initialize()`.

## Goal

Minimal amendment to v7 so LD can attach downstream via the MIP-0002
event pattern. Two additions:

1. `DividendMinted` event emitted inside `settle()` after the dividend mint.
2. `setLivingDividendAddress` multisig-gated setter for the recipient config
   (so we can point at LD without hardcoding at deploy time).

Everything else in v7 stays identical.

## Additions to state

```compact
// Nullable pointer to the Living Dividend contract.
// When unset, dividend slice mints to _dividendRecipient (placeholder treasury)
// exactly like v7. When set, the DividendMinted event fires for LD to consume.
export ledger _livingDividendAddress: Maybe<ContractAddress>;
```

Default is `none<ContractAddress>()` at deploy — v7.1 behaves exactly like
v7 until multisig sets the LD address. Zero-risk migration.

## Event struct

```compact
struct DividendMinted {
  sourceTxSalt: Bytes<32>;
  amount:       Uint<128>;
  recipient:    ContractAddress;
  blockTime:    Uint<64>;
  epochColor:   Bytes<32>;
}
```

## Additions to circuits

### `settle()` — two-line addition at the tail

After the existing dividend mint call, before settlement bookkeeping:

```compact
// Compute the uniqueness salt from sessionID (v5.2 replay-guard-unique)
const salt = persistentHash<Vector<3, Bytes<32>>>([
  pad(32, "pollpower:dividend:v7.1"),
  disclose(sessionID),
  disclose(currentTime as Field) as Bytes<32>,
]);

// Emit event iff LD is configured; otherwise v7.1 is functionally v7.
if (disclose(_livingDividendAddress.is_some)) {
  emit (DividendMinted {
    sourceTxSalt: disclose(salt),
    amount:       disclose(divAmt),
    recipient:    disclose(_livingDividendAddress.value),
    blockTime:    disclose(currentTime),
    epochColor:   tokenType(pad(32, "pollpower:ebt:v7:epoch1"), kernel.self()),
  });
}
```

### `setLivingDividendAddress` — new multisig-gated setter

```compact
export circuit setLivingDividendAddress(
  addr: ContractAddress,
  currentTime: Uint<64>,
): [] {
  assert(_initialized, "Not initialized");
  assert(disclose(witness_blockTimeGte(disclose(currentTime))),
         "setLDAddr: timestamp in future");

  // Multisig payload (ADR-002 pattern)
  const payload = persistentHash<Vector<3, Bytes<32>>>([
    pad(32, "ebt:setLDAddr:v7.1"),
    disclose(addr).bytes,
    disclose(currentTime as Field) as Bytes<32>,
  ]);
  assert(disclose(witness_multisigSignatureValid(payload, _multisigOwner)),
         "setLDAddr: multisig signature invalid");

  _livingDividendAddress = some<ContractAddress>(disclose(addr));
}
```

Optional companion `clearLivingDividendAddress()` circuit for emergency
disable — sets back to `none`. Same multisig pattern. Recommended.

## Deployment sequence

1. **Deploy LD contract** first, capture `ldContractAddress`.
2. **Deploy v7.1** with `_livingDividendAddress = none` — v7.1 is
   functionally identical to v7 until step 4.
3. **Migrate settlement-api config** from v7 to v7.1 endpoint.
4. **Verify v7.1 settle works end-to-end** with a test session (uses
   placeholder dividend recipient, same as v7).
5. **Multisig-set** `setLivingDividendAddress(ldContractAddress)` on v7.1.
   From this moment forward, every settle emits `DividendMinted` and
   sends the dividend slice to LD.
6. **Start keeper service** on relay (already listening; picks up events
   from step 5 onward).

Migration is atomic per step. If any step fails, roll back to prior step
without breaking the pilot's core settle path.

## What v7.1 does NOT do

- No `mintUnshieldedToken` to LD contract directly. The dividend slice
  still mints to `_dividendRecipient` (which we set to LD's ContractAddress
  when `_livingDividendAddress` is set). LD receives via the same v7 mint
  primitive; the event is a *notification*, not a value transfer.
- No cross-contract calls (still not supported in Compact).
- No changes to producer / operations / DAO mint logic.
- No changes to redeem, transfer, or any other v7 circuit.

## Risk assessment

**Very low.** The addition is:
- One nullable pointer field (default none)
- One conditional event emission (skipped when nullable is none)
- One new setter (multisig-gated, non-adversarial value type)

If v7.1 has a bug, we roll back to v7. Producers, consumers, ops, DAO
paths all unaffected. Worst case: keeper stops seeing events and LD
accruals pause temporarily. No funds at risk.
