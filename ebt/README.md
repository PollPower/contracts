# EBT — Energy-Backed Token

> **1 EBT = 1 kWh. Always. Immutably. Forever.**

EBT is the unit of account for the PollPower energy economy. One token represents one verified kilowatt-hour of renewable energy generated by a registered producer.

This directory contains two versions of the EBT contract:

| File | Status |
|---|---|
| [`ebt-v5.compact`](./ebt-v5.compact) | ✅ **PRODUCTION** — the contract currently minting EBT for live settlements |
| [`ebt-v5.1.compact`](./ebt-v5.1.compact) | 🟡 **DEPLOYED — cutover pending** — architectural successor; verifies hardware + Meter Authority signatures in-circuit |

## EBT v5 (production)

**Address (Midnight Preview):** `5cbc10a7a8f43a86fab8a8a015823b973be887b41bf6e2b03b51eb2dccff3b0e`
**Deployed:** 2026-05-04

### What it does

EBT v5 is the live mint path. Every kWh consumed at a PollPower metered outlet results in a `settle()` call that:

1. Verifies the caller is the registered **settlement authority** (the off-chain bridge run by PollPower)
2. Verifies the meter that recorded the consumption is in the on-chain meter registry
3. Mints EBT to the producer who delivered the energy, split according to the on-chain margin policy
4. Locks the entry rate (KES/kWh at the time of settlement) for later redemption

Key circuits:

| Circuit | Who calls | What it does |
|---|---|---|
| `settle()` | Settlement authority | Mint EBT to producer after consumption is proven |
| `redeemForKes()` | Pool authority or holder | Burn EBT, return locked entry rate for fiat payout |
| `transfer()` / `transferFrom()` | Any holder | Send EBT to another verified user |
| `registerMeter()` / `deregisterMeter()` | Owner (council-controlled) | Register a meter's public-key hash |

### Trust model

The settlement authority is currently a single key controlled by PollPower's settlement service. **However**, the inputs to `settle()` cannot be fabricated:

- The producer recipient must be in the [`ProducerRegistry`](../producer-registry/), which requires 3-of-5 council multi-sig signatures to modify.
- The meter ID must have been previously registered by the owner (also council-gated in production).

The settlement authority can *call* `settle()`, but cannot make `settle()` accept a fake producer or an unregistered meter.

## EBT v5.1 (deployed, cutover pending)

**Address (Midnight Preview):** `e3514ab0c5dca1a61700ac96f12f80157ea41474642161ce91cdd62dc0a1291d`
**Deployed:** 2026-05-09

### What changes from v5

v5.1 removes the privileged-caller requirement and replaces it with **two in-circuit signature verifications**:

1. **HAT signature** — the metering hardware must have signed the session payload with its private key. The key never leaves the chip.
2. **Meter Authority attestation** — the Meter Authority service must have signed an attestation that this specific gateway is currently approved, with an expiry timestamp.

The on-chain Meter Authority pubkey is set by the council (via `setMeterAuthority`), rotated only via multi-sig ceremony. The list of approved gateways lives off chain in the Meter Authority's database — no on-chain meter list at all.

This makes the mint path **permissionless** (anyone with a valid signature pair can submit `settle()`) while gating it on cryptographic proof of provenance.

### Why it isn't wired in production yet

Two prerequisites:

1. The v3.4 Gateway HAT (PollPower's metering hardware) must be in the field producing real signed readings. The HAT is currently in bring-up.
2. The on-chain Meter Authority pubkey on the v5.1 contract is currently set to a value that was committed via the pilot-mock 3-of-5 council ceremony (see [STATUS](../producer-registry/STATUS.md)). Real Tangem ring keys replace the pilot-mocks before cutover.

When both are done, the production settlement service will be updated to call v5.1's `settle()` instead of v5's. EBT v5 will remain on chain forever as the historical first-mint contract; v5.1 will become the live mint path.

See [`v5.1-DESIGN.md`](./v5.1-DESIGN.md) for the full architectural intent.

## Reading v5 → v5.1 together

The progression is the architectural story of the project:

- **v5:** privileged caller, but inputs cannot be faked
- **v5.1:** no privileged caller, every input cryptographically attested

In neither version can PollPower-the-entity unilaterally fabricate a mint. v5.1 makes that property visible in a single contract read: the source code says exactly which signatures are required.
