# EBT v5.2 Migration Guide

**From:** EBT v5.1 (deployed, unwired — dead-letter per audit)
**To:** EBT v5.2 (audit-hardened, staged for deploy ceremony)
**Date:** 2026-06-12
**Audit ref:** 2026-06-10 Fable 5 review, tag v10.0

## Breaking changes requiring off-chain coordination

### 1. HAT Signed Payload: 3-field → 4-field (C-1 fix)

**v5.1 payload:**
```
sessionPayloadHash = persistentHash<Vector<3, Bytes<32>>>([
  sessionID,
  hatPubkey,
  amountBytes
])
```

**v5.2 payload:**
```
sessionPayloadHash = persistentHash<Vector<4, Bytes<32>>>([
  sessionID,
  hatPubkey,
  amountBytes,
  producer          // NEW — recipient wallet address as Bytes<32>
])
```

**Who needs to change:**
- **HAT firmware** — must include `producer` in the signed payload
  before computing the Ed25519 signature. The producer address is
  static per HAT (provisioned at setup time). See HAT firmware spec below.
- **settlement-api** — `proverWitness.ts` must construct the 4-field
  hash when verifying HAT signatures off-chain.
- **Meter Authority Service** — NO change. Authority attestation still
  signs `(hatPubkey, expiresAt)` only.

### 2. settle() now requires attestationKey + meterKeyHash (M-1 fix)

v5.1 `settle()` took: sessionID, hatPubkey, amount, expiresAt, producer,
producerAmt, opsAmt, divAmt, daoAmt, hatSig, authoritySig.

v5.2 `settle()` adds two parameters after `producer`:
- `attestationKey: Bytes<32>` — must equal `persistentHash([producer, meterKeyHash])`
- `meterKeyHash: Bytes<32>` — the meter's key hash

**settlement-api change:** compute `attestationKey` from `producer` and
`meterKeyHash` before calling `settle()`. The contract will verify the
binding.

### 3. manualReissue now requires cap, cooldown, co-signature (M-4 fix)

v5.1/v5 had unconstrained `manualReissue(producer, amount, reasonCode, evidenceHash)`.

v5.2 requires:
- `amount <= 1,000,000` base units (1 EBT) per call
- `currentTime >= _lastReissueTime + 3600` seconds
- `authoritySig` — Ed25519 signature from Meter Authority over `hash(producer, amountBytes)`

**Admin tooling change:** batch large reissues into multiple calls
spaced ≥1 hour apart. Each call needs a co-signature from the Meter
Authority key.

### 4. attestProducerOwnership signature change (M-1 + L-3)

New parameters: `producerBytes`, `meterKeyHash`, `currentTime`.
Old parameters `attestationKey` must now match `hash(producerBytes, meterKeyHash)`.

## HAT Firmware Payload Spec (v5.2)

### Signed message format

```
// Fields (all 32 bytes each, in order):
field[0] = sessionID           // 32-byte session identifier
field[1] = hatPubkey            // 32-byte Ed25519 public key of the HAT
field[2] = amountBytes          // Uint<128> amount, cast to Field, then to Bytes<32>
field[3] = producerAddress      // 32-byte producer wallet address (static per HAT)

// Hash:
messageHash = Poseidon hash of [field[0], field[1], field[2], field[3]]
              (using Midnight's persistentHash<Vector<4, Bytes<32>>>)

// Signature:
signature = Ed25519.sign(hatPrivateKey, messageHash)
```

### Amount encoding

```c
// In firmware (C), amount is a uint64_t representing base units.
// Cast to 32-byte big-endian Bytes<32>:
uint8_t amount_bytes[32] = {0};
// amount_bytes[24..31] = amount as 8-byte big-endian
// Then Poseidon-hash the 4-field vector
```

**Note:** The exact Poseidon hash must match Midnight's `persistentHash`
implementation. Use the same library the prover witness uses
(`@midnight-ntwrk/compact-runtime`'s `persistentHash`). For embedded
C, this will need a Poseidon implementation matching Midnight's
parameters. **This is a known open work item for the firmware team.**

### Producer address provisioning

The producer address is static per HAT — set during initial provisioning.
Stored in the HAT's non-volatile config alongside the HAT private key
and session counter. Not rotatable without re-provisioning.

## Deploy ceremony checklist

1. [ ] Compile v5.2 with full ZK (no `--skip-zk`) on Kenya
2. [ ] Deploy to Preview network alongside v5 (keep v5 live)
3. [ ] Run `initialize()` with current Meter Authority pubkey
4. [ ] Update settlement-api to construct 4-field payload
5. [ ] Migrate one test producer: `attestProducerOwnership()` → `settle()` end-to-end
6. [ ] Verify settlement count, balance, replay protection
7. [ ] Cut over settlement-api to v5.2 as the active contract
8. [ ] Mark v5.1 as DEPRECATED in repo
9. [ ] DO NOT mark v5 as deprecated until v5.2 has run for several days

## What's NOT changing

- Meter Authority Service — no contract-side changes
- Consumer/producer app balance reads — same `balanceOf()` interface
- Settlement slice policy — same BPS defaults
- Network — staying on Preview
